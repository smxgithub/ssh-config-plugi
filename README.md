# Remote Config Generator

Edit Python locally in Trae (or VS Code), debug it on a remote machine over SSH using a remote conda env as the interpreter — PyCharm-style. Also generates SSH-aware rule files so AI agents (Claude Code, Trae's agent, Codex) drive the same remote workflow.

## What this extension does

One command (`Generate Remote Dev Configs`) sets up a workspace for a remote-execution loop:

- **For you (the human)**: a status bar button **`▶ Debug on Remote`** that syncs the active file to the remote, starts `debugpy` over SSH with auto-install, forwards the debug port, and attaches Trae's debugger automatically. Set a breakpoint and click — the breakpoint hits.
- **For agents**: rule files (`.claude/rules/sshrule.md`, `.trae/rules/sshrule.md`, `AGENTS.md`) tell agents to edit locally only and run/test on the remote via SSH.
- **For sync**: `.vscode/sync-remote.ps1` and `.vscode/sftp.json` for deterministic and on-save uploads.

## Files generated by the wizard

In `.vscode/`:
- `sftp.json` — VS Code SFTP extension config (upload-on-save).
- `sync-remote.ps1` — deterministic sync helper for touched files.
- `remote-debug.ps1` — orchestrates ssh + debugpy on the remote.
- `remote-config-gen.json` — machine-readable config the plugin reads at debug time.
- `launch.json` — debug attach configuration (`Run on Remote (Python)`).
- `tasks.json` — `remote-sync`, `remote-run`, `remote-debug-launch`, `remote-install-debugpy`.

In the workspace root and `.claude/`/`.trae/`:
- `AGENTS.md`, `.claude/rules/sshrule.md`, `.trae/rules/sshrule.md` — agent rule files.

The plugin's merge logic preserves your own entries in `launch.json`/`tasks.json` and only replaces what it owns by name.

## Setup (once per workspace)

1. Install the `.vsix` from VSIX (or from a marketplace if published).
2. Open your workspace folder.
3. `Ctrl+Shift+P` → **"Generate Remote Dev Configs"**.
4. Walk through the 4-step wizard:
   - Path to your SSH config file (default `~/.ssh/config`).
   - Pick the SSH host alias (parsed from your config).
   - Type the absolute project path on the remote (e.g. `/home/you/myproject`).
   - Pick the conda env. The plugin SSHs into the host and auto-detects available envs; if detection fails, fall back to typing the env name and conda root path manually.

That's it. All the files above are generated.

## Daily usage — debugging a Python file

1. Open the `.py` file you want to debug. Make sure its tab is the active editor.
2. Click in the gutter (left of the line number) to set a **breakpoint** — a red dot appears.
3. Click **`▶ Debug on Remote`** in the status bar (bottom-left).

The plugin will:

1. Sync your active file to the remote.
2. SSH to the remote, source conda init, activate the env.
3. Auto-install `debugpy` on first run if missing.
4. Kill any leftover debugpy on the debug port (zombie cleanup).
5. Start `python -m debugpy ... --wait-for-client your_file.py`.
6. Wait for `debugpy` to announce itself, then attach Trae's debugger.

Your breakpoint hits. Step (`F10`/`F11`), inspect variables in the Variables panel, evaluate in the Debug Console, continue (`F5`), or stop (red square).

## Daily usage — running without the debugger

`Ctrl+Shift+P` → **"Tasks: Run Task"** → **`remote-run`**.

Output streams to a Trae terminal. The active file is synced, then run on the remote in your conda env.

## Troubleshooting

- **Output channel `Remote Debug`** has every step the plugin took, prefixed with `[plugin]`, `[sync]`, `[ssh-out]`, `[ssh-err]`. Open it via View → Output → "Remote Debug".
- **`.vscode/remote-debug.log`** has timestamped diagnostic logs from `remote-debug.ps1` (only relevant if you invoke that script directly; the plugin's normal flow uses the Output channel).
- **Conda env detection failed** → wizard falls back to typing the env name + conda root path. Default conda root: `~/miniconda3`. If yours is `~/anaconda3` or `/opt/conda`, type that.
- **Port 5678 already in use on remote** → 0.4+ auto-cleans this on every run via `lsof -ti tcp:5678 | xargs -r kill`. If you saw this on an older version, the issue is now solved.
- **`F5` alone doesn't work** → that's expected. F5 uses the launch config in pure-attach mode and expects `debugpy` to already be listening. Use the status bar button instead — it does the launch + attach in one step.
- **Want to override the debug port?** → edit `.vscode/launch.json` (`connect.port`) and `.vscode/remote-config-gen.json` (`debugPort`). Defaults to `5678`.

## How it works under the hood

PyCharm's "Remote Interpreter" + "Remote Debug" features are paid Pro. Trae and VS Code don't ship an equivalent out of the box. This extension stitches together the components that *are* available:

- `debugpy` (Microsoft's Python debug adapter) on the remote, started via SSH with `--wait-for-client`.
- `ssh -L <port>:localhost:<port>` to forward the debugger socket through SSH (no firewall holes needed).
- `vscode.debug.startDebugging` to attach Trae's existing Python debugger to the forwarded port.
- A `remote-config-gen.json` config the plugin reads to know your host alias, conda paths, port, etc., without parsing the generated PowerShell scripts.

The result is a one-click PyCharm-style remote interpreter experience inside Trae.

## Compatibility

- **Local OS**: Windows, macOS, and Linux. The plugin calls `ssh`/`scp` directly (available on all modern OSes). Platform-specific shell scripts (`.ps1` for Windows, `.sh` for macOS/Linux) are generated for manual task usage.
- **Remote OS**: any Linux with `bash`, `ssh`, `lsof`, and a conda installation.
- **Trae**: any version that's a recent VS Code fork (debug API + `debugpy` extension required).
- **VS Code**: works the same; uses the `ms-python.debugpy` adapter type.
